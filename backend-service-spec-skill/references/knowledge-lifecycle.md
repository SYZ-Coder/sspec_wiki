# 知识生命周期规则

本规则用于管理 `mydocs/` 中的知识产物，不再把所有梳理都当成一次性输出。

## 1. 三类知识状态

### 1.1 current

表示当前稳定、可供团队复用的知识版本。

典型位置：

- `mydocs/services/<service>/`
- `mydocs/domains/<domain>/`
- `mydocs/index/`

特点：

- 已通过基本质量检查
- 适合被后续增量更新
- 可以作为后续梳理的首选读取入口

### 1.2 draft

表示本轮尚未完成、尚未放行的过程产物。

典型位置：

- `mydocs/codemap/`
- `mydocs/routermap/`
- `mydocs/context/`
- `mydocs/specs/`
- `mydocs/validation/`

特点：

- 可以帮助继续推进梳理
- 不能默认视为最终稳定知识页
- 若内容未闭环，应保留边界和不确定性

### 1.3 archive

表示历史版本、旧轮次知识、过时但仍有追溯价值的内容。

特点：

- 不作为默认优先读取入口
- 只在需要比较历史差异、追溯演变时使用
- 不应与 current 混淆

## 2. 默认读取顺序

在重新梳理前，优先按以下顺序读取：

1. 先读 `mydocs/services/`、`mydocs/domains/`、`mydocs/index/` 下 current 稳定页
2. 再读与当前目标最相关的 `mydocs/` draft / 过程页
3. 最后才回到源码重新全量扫描

原则：

- 已有稳定知识优先
- 增量补齐优先
- 全量重扫最后考虑

## 3. 默认更新策略

### 3.1 如果稳定页已存在

- 优先补缺，不优先重写
- 只在结构明显失真、范围错误或证据等级过粗时考虑重建

### 3.2 如果只有 draft 存在

- 先判断当前 draft 是否还能继续使用
- 能继续时优先补证、补页、补边界
- 不能继续时再废弃并新开一轮

### 3.3 如果 current 与 draft 同时存在

- current 继续作为默认读取入口
- draft 作为本轮工作的补充上下文
- 不允许 draft 自动覆盖 current，除非已完成质量检查并明确替换

## 4. 何时允许重建

满足以下任一情况时，可以考虑整块重建：

- scope 判断错误，导致旧文档从根上分层错误
- 关键链路已发生结构性变化，旧文档大面积失真
- 稳定页长期未维护，无法再增量修补
- 旧文档没有来源追溯，已不适合作为 current

## 5. 放行规则

只有满足以下条件的内容，才适合从 draft 提升为 current：

- 范围明确
- 来源可追溯
- 事实边界明确
- 分层定位清楚
- 有团队复用价值

如果以上条件未满足，则继续保留在 draft 层。

## 6. 禁止事项

- 不要把 `mydocs/codemap/`、`mydocs/routermap/` 过程页直接当成稳定知识正文
- 不要每次都从源码零开始，忽略已有稳定页
- 不要让 draft 自动覆盖 current
- 不要把 archive 当成当前事实入口
