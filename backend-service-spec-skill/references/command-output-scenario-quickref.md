# Backend Skill Quick Reference

This page helps users quickly decide:

- which command to use
- what it usually produces
- which layer inside `mydocs/` should store the outputs
- what scenarios it fits best

Default rule:

- when a core command is executed as a standard output run, its companion diagram artifacts should also be generated by default
- only omit diagrams when the scope is extremely small, evidence is too weak to draw a responsible diagram, or the user explicitly asks for text-only output

## `create_codemap`

Typical outputs:

- service inventory page
- service relationship overview
- technology/module layering page
- upstream/downstream summary
- architecture diagram in Markdown + Mermaid
- service call graph in Markdown + Mermaid

Recommended directory:

```text
mydocs/codemap/
```

Companion diagram directories:

```text
mydocs/diagrams/architecture/
mydocs/diagrams/call-graph/
```

Best for:

- first-time system entry
- getting the overall structure first
- identifying high-value services

## `service_deep_dive`

Typical outputs:

- single-service structure page
- interface inventory page
- module partition page
- dependency page
- service rules/spec page
- upstream/downstream dependency diagram in Markdown + Mermaid
- internal module architecture diagram in Markdown + Mermaid

Recommended directory:

```text
mydocs/services/
```

Companion diagram directories:

```text
mydocs/diagrams/upstream-downstream/
mydocs/diagrams/architecture/
```

Best for:

- high-value service analysis
- understanding one service deeply
- preparing later route or domain work

## `crate_router_map`

Typical outputs:

- key route page
- synchronous route page
- asynchronous route page
- retry/compensation page
- closure status page
- sequence diagram in Markdown + Mermaid
- route call graph in Markdown + Mermaid

Recommended directory:

```text
mydocs/routermap/
```

Companion diagram directories:

```text
mydocs/diagrams/sequence/
mydocs/diagrams/call-graph/
```

Best for:

- tracing one real request
- tracing one message flow
- separating sync / async / compensation paths

## `build_domain_map`

Typical outputs:

- domain page
- domain -> service -> spec mapping page
- domain route summary
- domain rules page
- domain context diagram in Markdown + Mermaid

Recommended directory:

```text
mydocs/domains/
```

Companion diagram directory:

```text
mydocs/diagrams/architecture/
```

Best for:

- team central knowledge repositories
- cross-service domain abstraction
- lifting service facts into reusable knowledge pages

## Lightweight Full

Typical output set:

- scope page
- codemap overview
- 1 to 2 service deep dives
- 1 to 2 key route pages
- index page
- companion diagrams for the codemap, selected services, and selected routes by default

Recommended directory:

```text
mydocs/
  codemap/
  services/
  routermap/
  diagrams/
```

Best for:

- first pass on a project
- lower token cost
- building a reliable first baseline

## Heavy Full

Typical output set:

- scope page
- codemap overview
- multiple service deep dives
- multiple key route pages
- domain page
- validation page
- unresolved list
- index page
- companion diagrams for codemap, services, routes, and domains by default

Recommended directory:

```text
mydocs/
  codemap/
  services/
  domains/
  routermap/
  diagrams/
  context/
  specs/
  validation/
  index/
```

Best for:

- core system knowledge capture
- central knowledge repository building
- building a more complete long-term baseline
